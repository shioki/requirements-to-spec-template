# 要求仕様書テンプレート

## ID凡例とプレフィックス

| プレフィックス | 日本語名 | 説明 |
| --- | --- | --- |
| `業務-XX` | 業務要求 | 達成したいビジネス状態・目的 |
| `利用-XX` | 利用要求(ユースケース) | 利用者が行いたい行動・シナリオ |
| `機能-XX` | 機能要求 | 条件成立時のシステム動作 |
| `非機能-XX` | 非機能要求 | 性能・可用性・セキュリティ等の品質属性 |
| `制約-XX` | 制約条件 | 守るべき制限(法令・技術・契約など) |
| `連携-XX` | 外部連携要求 | 外部システム・API・ファイル連携 |
| `前提-XX` | 前提・依存 | 前提条件と外部依存タスク |
| `未解決-XX` | 未解決事項 | 本文に書かず切り出した未確定事項 |
| `データ-XX` | データ要求 | データ項目・整合性・保持・マスタ参照 |
| `リスク-XX` | リスク | 影響度・発生確率と対応方針 |

**優先度**: `必須` / `推奨` / `任意`  
**ドキュメントID**: `要件-XXXX` 形式で付番する。

## 記入の流れ(参照図)

```mermaid
flowchart TD
  docControl["ドキュメント管理"] --> projectOverview["1. プロジェクト概要"]
  projectOverview --> stakeholderAnalysis["2. ステークホルダー分析"]
  stakeholderAnalysis --> businessReq["3. 業務要求"]
  businessReq --> userCases["4. 利用要求"]
  userCases --> functionalReq["5. 機能要求"]
  functionalReq --> nonFunctionalReq["6. 非機能要求"]
  nonFunctionalReq --> constraints["7. 制約条件"]
  constraints --> interfaces["8. 外部連携要求"]
  interfaces --> assumptions["9. 前提・依存"]
  assumptions --> openIssues["10. 未解決事項"]
  openIssues --> traceability["11. トレーサビリティ"]
  traceability --> dataReq["12. データ要求"]
  dataReq --> risks["13. リスク"]
  risks --> glossary["14. 用語定義"]
```

## レビュー運用シーケンス(参照図)

```mermaid
sequenceDiagram
  actor Author as 作成者
  actor Reviewer as レビュアー
  actor Approver as 承認者
  participant Spec as 要求仕様書
  participant OpenIssues as 未解決一覧

  Author->>Spec: 初版を作成
  Reviewer->>Spec: 業務/利用/機能/非機能をレビュー
  Reviewer-->>Author: 修正コメント
  Author->>OpenIssues: 未確定事項を登録
  Author->>Spec: 修正反映版を更新
  Approver->>Spec: 最終確認
  Approver-->>Author: 承認/差戻し
```

## 5. 機能要求

### 記述ルール

- 推奨形式: 「[条件/トリガー] のとき、システムは [動作] する」
- あいまい語(なるべく、適切に、可能な限り)を避ける
- 実装手段ではなく期待するシステム動作を書く

### 5.x 受け入れ基準(各機能に対して記入)

チェックリスト(条件・入力・期待結果・異常系・境界値)に加え、必要に応じて **Given-When-Then** で補足する。

**Given-When-Then の例**

```text
Given  有効なユーザーアカウントが存在する
When   ユーザーが正しいIDとパスワードでログインを試みる
Then   システムはダッシュボードへ遷移し、最終ログイン日時を更新する
```
